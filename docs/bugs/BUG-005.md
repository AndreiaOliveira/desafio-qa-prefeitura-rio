# BUG-005 — GET /api/v1/services/:id/recommendations no grupo público, sem exigir auth

**Severidade:** Alta (segurança e consistência de controle de acesso)
**Endpoint:** GET /api/v1/services/:id/recommendations
**Código-fonte:** api/main.go, linhas 242 e 321
**Status:** Reproduzível e coberto por teste automatizado

## O que acontece

O endpoint de recomendações está registrado no grupo de rotas públicas, embora o design da API coloque outras operações relacionadas a preferências do usuário no grupo protegido (como POST /services/:id/favorite).

Comentários no código confirmam:
- "registered in the public route group — no auth check performed"
- "this endpoint should be in the protected group below"

## O que deveria acontecer

Recommendations deveria estar no grupo protected junto com favorite, exigindo Authorization: Bearer <token>:

- Sem token: 401 Unauthorized com {"error":"missing authorization header"}
- Token inválido: 401 Unauthorized com {"error":"invalid token"}
- Token válido: 200 OK com as recomendações

## Como reproduzir

Sem qualquer autenticação, a API retorna os dados normalmente:

    curl -i http://localhost:8080/api/v1/services/s001/recommendations

Resposta observada: 200 OK com JSON de recommendations.

Comparando com o comportamento esperado (igual ao favorite):

    curl -i -X POST http://localhost:8080/api/v1/services/s001/favorite

Resposta: 401 Unauthorized — {"error":"missing authorization header"}.

Ou seja, dois endpoints do mesmo recurso tratam auth de forma diferente, contrariando o contrato implícito.

## Impacto

- Controle de acesso inconsistente: consumidores da API não conseguem inferir com segurança quais endpoints exigem auth. Isso complica a implementação de clientes e de documentação.
- Enumeração: recommendations podem revelar padrões de associação entre serviços (quais se relacionam com quais), que num contexto real podem ser informação sensível para derivar perfis de uso ou alimentar engenharia social.
- Limite de taxa / abuso: endpoints públicos são mais expostos a scraping e abuso. Se esse endpoint no futuro expor dados pessoais ou métricas agregadas de comportamento, o risco cresce proporcionalmente.
- Conformidade com o próprio design: a presença do grupo protected mostra que o time pretende separar operações autenticadas das não-autenticadas; esse endpoint está do lado errado da linha por acidente.

## Hipótese de correção

Mover o registro da rota do grupo public para o grupo protected em api/main.go (perto da linha 331, junto com favorite):

    protected.GET("/services/:id/recommendations", handleRecommendations)

E remover o registro correspondente do grupo public.

## Teste automatizado

Coberto por `tests/api/recommendations/auth.spec.ts`:

- `returns 200 with a valid token` — happy path (passa hoje)
- `rejects request without Authorization header (BUG-005a)` — `test.fail`
- `rejects request with invalid token (BUG-005b)` — `test.fail`

Quando o endpoint for movido para o grupo `protected`, os dois `test.fail` passarão a falhar e devem ser convertidos em `test()` regular. O happy path com token válido continua validando que a operação funciona — protege contra correção mal feita que rejeite tudo.
