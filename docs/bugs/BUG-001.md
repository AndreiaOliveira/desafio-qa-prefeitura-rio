# BUG-001 — GET /api/v1/services/:id retorna 500 em vez de 404 para ID inexistente

**Severidade:** Alta
**Endpoint:** GET /api/v1/services/:id
**Código-fonte:** api/main.go, linhas 201 e 205
**Status:** Reproduzível e coberto por teste automatizado

## O que acontece

Quando o cliente requisita o detalhe de um serviço com ID que não existe no catálogo, a API retorna HTTP 500 Internal Server Error com corpo vazio, em vez de 404 Not Found.

Comentário no código: "no nil check before dereferencing. When service is nil, nil pointer dereference."

## O que deveria acontecer

404 Not Found com JSON no padrão já usado pela API:

    {"error":"service not found"}

## Como reproduzir

    curl -i http://localhost:8080/api/v1/services/nao-existe

Saída observada: HTTP/1.1 500 Internal Server Error (body vazio)

## Impacto

- Monitoramento: 500s disparam alertas desnecessariamente
- Usuário final: cliente HTTP interpreta como servidor com problema, não como "serviço não encontrado"
- Contrato quebrado: outros endpoints retornam 404 JSON (ex: /recommendations)
- Estabilidade: nil pointer panic indica código frágil

## Hipótese de correção

Adicionar verificação nil antes do dereferência e retornar 404 explicitamente.

## Teste automatizado

Coberto por:

- `tests/api/services/detail.spec.ts` → `returns 404 for a non-existent service id (BUG-001)` (marcado `test.fail` até a correção)
- `tests/api/security/negative-cases.spec.ts` → 5 variantes adicionais marcadas `test.fail` (ver abaixo)

## Variantes adicionais descobertas

Durante exploração com payloads adversários, descobri que o crash não se limita a IDs "inexistentes" simples. Os mesmos sintomas (HTTP 500 sem body, provavelmente nil-pointer panic) aparecem para:

| Payload | Tipo | Spec |
|---|---|---|
| `s001'; DROP TABLE--` | SQL-like injection | `negative-cases.spec.ts` |
| `..%2F..%2Fetc%2Fpasswd` | Path traversal url-encoded | `negative-cases.spec.ts` |
| `aaaa...` (5000 chars) | ID muito longo | `negative-cases.spec.ts` |
| `ѕ001` (s cirílico) | Unicode homoglyph | `negative-cases.spec.ts` |
| `   ` | Whitespace puro | `negative-cases.spec.ts` |

Curiosamente, payloads de XSS literal (`<script>alert(1)</script>`) e path traversal unix (`../../etc/passwd`) **passam ilesos** — provavelmente porque o roteador Gin os rejeita antes de chegar ao handler. Esse comportamento inconsistente é mais perigoso do que o bug original: dá uma falsa sensação de que algumas defesas funcionam.

A correção do BUG-001 (verificação nil + 404 explícito) deve resolver todas as variantes ao mesmo tempo, já que a causa raiz é a mesma.
