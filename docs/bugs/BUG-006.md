# BUG-006 — recommendations retorna null em vez de array vazio

**Severidade:** Média
**Endpoint:** GET /api/v1/services/:id/recommendations
**Status:** Reproduzível e coberto por teste automatizado
**Origem:** Achado durante desenvolvimento da suite de contract testing (não estava na lista de bugs plantados)

## O que acontece

Quando o serviço requisitado não possui recomendações associadas, a API retorna o campo `recommendations` com valor `null` em vez de array vazio `[]`. O contrato em outros endpoints (e o que consumidores razoavelmente esperam) é que listas sempre sejam arrays, mesmo quando vazias.

Comparativo:

| Serviço | Possui recomendações | Valor de `recommendations` |
|---|---|---|
| s001 | Sim (1 item) | `[ {...} ]` (array) |
| s005 | Não | `null` |
| s011 | Sim (1 item) | `[ {...} ]` (array) |

## O que deveria acontecer

A API deveria retornar sempre um array, vazio quando aplicável:

    {
      "recommendations": [],
      "service_id": "s005"
    }

Esse é o padrão consistente: campo de tipo lista permanece sendo lista, independentemente de ter ou não conteúdo.

## Como reproduzir

    curl -s http://localhost:8080/api/v1/services/s005/recommendations | python3 -m json.tool

Resposta atual:

    {
        "recommendations": null,
        "service_id": "s005"
    }

Para confirmar que o comportamento é inconsistente entre serviços:

    curl -s http://localhost:8080/api/v1/services/s001/recommendations | python3 -m json.tool

Aqui retorna array com 1 item.

## Impacto

- **Crash de cliente**: código JavaScript que faz `body.recommendations.map(...)` ou `body.recommendations.length` lança `TypeError: Cannot read properties of null` quando aponta para um serviço sem recomendações. Isso quebra renderização em produção sem aviso.
- **Fricção desnecessária**: todo cliente que consome esse endpoint precisa adicionar guard defensivo `(body.recommendations || []).map(...)` por causa de uma inconsistência de servidor que poderia ser corrigida na origem.
- **Inconsistência interna**: o endpoint GET /api/v1/services retorna `data: []` quando filtros não casam (não retorna `null`). Ter dois padrões diferentes na mesma API confunde integradores.
- **Contract violation real**: este é um exemplo perfeito de defeito que testes funcionais tradicionais não pegam (status 200, payload existe, parser não quebra) mas que afeta consumidores de verdade.

## Hipótese de correção

Provavelmente o handler em Go está retornando uma slice não-inicializada (que serializa para JSON null), em vez de uma slice explicitamente vazia (que serializa para `[]`).

Correção típica em Go:

    recs := []Service{}                  // explicit empty slice -> serializes as []
    // em vez de:
    var recs []Service                   // nil slice -> serializes as null

## Como foi descoberto

Durante a construção do spec de contract testing `tests/contract/recommendations-schema.spec.ts`, escrevi um teste validando que cada recommendation segue o schema completo de Service. O teste falhou ao tentar iterar sobre `body.recommendations` para o serviço s005. Investigação manual com `curl` confirmou que o campo retorna null, não array.

Isso ilustra o valor do contract testing: invariantes estruturais detectam comportamentos que o consumidor depende, mas que testes de happy path não exercitam (porque happy path não cobre o caso lista vazia).

## Teste automatizado

Coberto por:

- `tests/contract/recommendations-schema.spec.ts` -> `recommendations deve ser array vazio quando nao ha recomendacoes (BUG-006 - achado)` (marcado test.fail até a correção)

Quando o bug for corrigido, esse teste passa a falhar (porque test.fail espera falha) e o time saberá que pode remover a marcação.
