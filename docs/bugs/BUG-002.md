# BUG-002 — POST /api/v1/services/search com query vazia retorna todos os resultados

**Severidade:** Média
**Endpoint:** POST /api/v1/services/search
**Código-fonte:** api/main.go, linhas 217 e 226
**Status:** Reproduzível e coberto por teste automatizado

## O que acontece

A busca aceita payload com campo query vazio ("") ou ausente e retorna 200 OK com os 11 serviços do catálogo, em vez de rejeitar a requisição.

Comentários no código:
- "empty query returns 200 with all results instead of 400"
- "missing empty query validation"

## O que deveria acontecer

400 Bad Request com mensagem explicativa:

    {"error":"query is required and must not be empty"}

A API já rejeita outros formatos inválidos (JSON malformado, tipo errado) — falta apenas validar conteúdo vazio.

## Como reproduzir

Caso 1 — query explicitamente vazia:

    curl -i -H "Authorization: Bearer qa-challenge-token" \
      -H "Content-Type: application/json" \
      -X POST -d '{"query":""}' \
      http://localhost:8080/api/v1/services/search

Caso 2 — campo query ausente:

    curl -i -H "Authorization: Bearer qa-challenge-token" \
      -H "Content-Type: application/json" \
      -X POST -d '{}' \
      http://localhost:8080/api/v1/services/search

Nos dois casos a API retorna 200 OK com os 11 serviços.

## Impacto

- Performance e DoS: query vazia força a API a percorrer e serializar o dataset completo. Em escala municipal com picos de notícia, repetir essa chamada vira vetor de consumo de recursos.
- Contrato quebrado: endpoint é uma busca — retornar tudo quando não se pede nada confunde o consumidor.
- UX: front-ends podem acidentalmente enviar query vazia (campo não preenchido, debounce mal feito) e mostrar "tudo" em vez de pedir para o usuário digitar.

## Hipótese de correção

Validar antes de buscar:

    if strings.TrimSpace(req.Query) == "" {
        c.JSON(400, gin.H{"error": "query is required and must not be empty"})
        return
    }

## Teste automatizado

Coberto por `tests/api/search/validation.spec.ts`:

- `rejects empty query string with 400 (BUG-002a)` — query explícita `""`
- `rejects missing query field with 400 (BUG-002b)` — campo `query` ausente

Ambos marcados `test.fail` até a correção. Quando o bug for corrigido, os testes passarão a falhar (porque `test.fail` espera falha), sinalizando que devem ser convertidos em `test()` regular.
