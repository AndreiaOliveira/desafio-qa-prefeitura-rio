# BUG-004 — Webhook lê a assinatura HMAC mas nunca valida

**Severidade:** Crítica (segurança)
**Endpoint:** POST /api/v1/webhooks/catalog
**Código-fonte:** api/main.go, linhas 264-270 e 286
**Status:** Reproduzível e coberto por teste automatizado

## O que acontece

O endpoint de webhook recebe o header X-Signature-256 (HMAC-SHA256 assinando o corpo da requisição com o secret webhook-secret-2024), mas a validação nunca é executada. O valor é apenas lido, logado com indicação de presença, e descartado. Qualquer requisição é aceita e processada com status 200.

Existe a função validateHMAC implementada no mesmo arquivo, porém ela não é chamada em lugar nenhum. Comentários no código confirmam:
- "reads the HMAC signature header but never validates it"
- "signature is read but immediately discarded"
- "The correct implementation should call validateHMAC and return 401 on failure"
- "This is the correct implementation — it is intentionally not called"

## O que deveria acontecer

Webhook deve validar a assinatura HMAC antes de processar o corpo. Respostas esperadas:

- Sem header X-Signature-256: 401 Unauthorized com {"error":"missing signature"}
- Com assinatura inválida: 401 Unauthorized com {"error":"invalid signature"}
- Com assinatura válida: 200 OK e processamento do payload

## Como reproduzir

Caso 1 — sem assinatura alguma:

    curl -i -H "Content-Type: application/json" -X POST \
      -d '{"event":"test"}' \
      http://localhost:8080/api/v1/webhooks/catalog

Resposta: 200 OK — {"received":"...","status":"accepted"}

Caso 2 — assinatura claramente inválida:

    curl -i -H "Content-Type: application/json" \
      -H "X-Signature-256: sha256=abc123" \
      -X POST -d '{"event":"test"}' \
      http://localhost:8080/api/v1/webhooks/catalog

Resposta: 200 OK — {"received":"...","status":"accepted"}

Em ambos os casos a API aceita a requisição sem qualquer verificação de autenticidade.

## Impacto

Este é um bug de segurança crítico. Consequências práticas:

- Injeção de dados externos: qualquer ator na internet que descobrir a URL do webhook pode disparar eventos falsos sem precisar conhecer o secret. Se eventos do webhook futuramente atualizarem o catálogo, isso permite manipulação do catálogo público.
- Secret inútil: o propósito da HMAC é garantir autenticidade e integridade (só quem tem o secret pode produzir assinaturas válidas). Sem validação, o secret não protege absolutamente nada.
- Superfície de ataque: em serviço público municipal, URL acaba vazando em logs, screenshots, exemplos de documentação, backups. A defesa em profundidade supõe que a URL é descobrível e que a HMAC é a real camada de proteção.
- Trilha de auditoria corrompida: eventos aceitos como legítimos podem ter sido forjados, dificultando investigação em incidente real.
- Conformidade: processamento de dados da população sem autenticação de origem pode ser questionado em auditorias (LGPD, controles internos).

Em um cenário real, este bug justificaria bloqueio de deploy em produção até ser corrigido.

## Hipótese de correção

O código já tem a função pronta. Basta chamá-la no handler:

    body, err := io.ReadAll(c.Request.Body)
    if err != nil {
        c.JSON(400, gin.H{"error": "invalid body"})
        return
    }
    signature := c.GetHeader("X-Signature-256")
    if signature == "" {
        c.JSON(401, gin.H{"error": "missing signature"})
        return
    }
    if !validateHMAC(body, signature, webhookSecret) {
        c.JSON(401, gin.H{"error": "invalid signature"})
        return
    }
    // continuar com o processamento

Cuidados adicionais:
- Usar hmac.Equal (constant-time compare) para evitar timing attacks — a função validateHMAC existente já faz isso
- Restaurar o body (c.Request.Body = io.NopCloser(bytes.NewBuffer(body))) se outros handlers precisarem ler depois

## Teste automatizado

Coberto por `tests/api/webhook/security.spec.ts`:

- `rejects request without X-Signature-256 header (BUG-004a)` — `test.fail`
- `rejects request with garbage signature (BUG-004b)` — `test.fail`
- `rejects request signed with wrong secret (BUG-004c)` — `test.fail`
- `accepts request with valid HMAC-SHA256 signature` — passa hoje (proteção contra correção destrutiva)

O helper `utils/hmac.ts` encapsula a geração de assinatura HMAC-SHA256 usando o módulo `node:crypto` nativo (zero dependência externa), garantindo que o cliente de teste assina exatamente como a API espera.
